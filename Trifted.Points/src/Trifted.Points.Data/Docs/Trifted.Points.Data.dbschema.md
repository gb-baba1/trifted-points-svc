## Do / Don't Guidance

| | Do | Don't |
| :---: | --- | --- |
| ✅ | Use `InsertWithIndexAsync` when the entity has auto-apply indexes | Use `InsertAsync` and manually manage index records — they will be out of sync |
| ✅ | Pass the **previous entity** to `UpdateWithIndexAsync` | Call `UpdateWithIndexAsync` with a freshly constructed entity — stale index records will remain |
| ✅ | Use the generated `Find*Async` methods for queries | Write raw DynamoDB `QueryRequest` calls — you lose key composition, pagination, and scatter-gather |
| ✅ | Use `GetItemCollectionAsync` to fetch related entities in one call | Issue separate `GetItem` calls for each entity in an item collection |
| ✅ | Let the generator compose keys via `Apply{Entity}KeyTemplates` or `Compose*` methods | Manually construct key strings — drift causes key mismatches and missing data |
| ✅ | Use `QueryConfigOption` fluent methods for filters and projections | Build raw `FilterExpression` strings — error-prone and not type-safe |
| ✅ | Use `isConsistentRead = false` for read-heavy, latency-tolerant paths | Default to strongly consistent reads everywhere — it doubles RCU cost |
| ✅ | Add `[Unique]` / `[CompositeUnique]` for uniqueness constraints | Check uniqueness via `Scan` — full table scan, expensive and slow |
| ✅ | Wrap transactional writes in `BeginTransaction()` … `CommitAsync()` | Call transactional methods outside a transaction — the analyzer flags this |
| ✅ | Use `[SearchableIndex]` for text-based lookups | Implement client-side string matching on full scans |

## GSI Storage Amplification

<details>
<summary>View GSI storage cost multiplier analysis</summary>

> Every GSI maintains its own copy of the projected attributes. An `ALL` projection replicates
> the entire item — effectively doubling storage per GSI. This is the **hidden cost multiplier**
> that catches teams off guard during cost reviews.

| Entity | GSI | Projection | Est. Entity Size | Est. GSI Item Size | Storage Multiplier |
| --- | --- | --- | --- | --- | --- |
| `AbstractBaseEntity` | `Gsi1Index` | `ALL` (assumed) | ~200 B | ~200 B | **×2** (base + GSI) |
| | | `KEYS_ONLY` (if changed) | ~200 B | ~218 B | ×2.1 |
| `AbstractBaseEntity` | `Gsi2Index` | `ALL` (assumed) | ~200 B | ~200 B | **×2** (base + GSI) |
| | | `KEYS_ONLY` (if changed) | ~200 B | ~218 B | ×2.1 |

> **Worst-case total storage multiplier:** With 2 GSIs using `ALL` projection, your effective storage is **3×** the base table size. At $0.25/GB-month, a 10 GB table costs **$7.50/month** in storage alone.

### GSI Projection Optimization Checklist

| Question | If Yes | If No |
| --- | --- | --- |
| Do queries on this GSI always fetch the full entity? | Use `ALL` (avoid read-back) | Use `INCLUDE` with only needed attrs |
| Is this GSI used only for existence checks or key lookups? | Use `KEYS_ONLY` | — |
| Is the entity large (>1 KB)? | `KEYS_ONLY` or `INCLUDE` saves significant storage | `ALL` is fine — marginal cost |
| Is the GSI write-heavy (high WAF)? | Minimize projection to reduce write amplification | — |

## Analyzer Rules Reference

The Kanject DynamoDB source generator ships with Roslyn analyzers that catch common mistakes at compile time.
Treat **Error**-level rules as hard blockers — build will fail. **Warning** and **Info** rules are advisory.

| ID | Severity | Category | Rule | Fix |
| --- | :---: | --- | --- | --- |
| `KAN001` | Error | Design | GSI alias must reference a valid `[DynamoDbGsi]` in the class hierarchy | Add a matching `[DynamoDbGsi(Name = "…")]` on the entity or a base type |
| `KAN002` | Error | Design | Duplicate GSI alias configurations on the same type | Remove the duplicate `[DynamoDbGsiAlias]` attribute |
| `KAN003` | Error | Design | Entity type in `EntityRepositoryAttribute<T>` must implement `IDynamoDbEntity` | Ensure `T` (or inner `T` in `List<T>`) implements `IDynamoDbEntity` |
| `KAN004` | Info | Modernization | Prefer generic `EntityRepository<T>()` over `EntityRepository(typeof(T))` | Replace `EntityRepository(typeof(Foo))` with `EntityRepository<List<Foo>>()` |
| `KAN005` | Info | Modernization | Prefer generic `EmbedRepository<T>()` over `EmbedRepository(typeof(T))` | Replace `EmbedRepository(typeof(Foo))` with `EmbedRepository<Foo>()` |
| `KAN006` | Error | Design | Array types (`T[]`) not supported in `EntityRepositoryAttribute<T>` | Use `List<T>` instead of `T[]` |
| `KANJECT001` | Error | Usage | Key template placeholder `{Prop}` references a non-existent property | Fix the property name — the analyzer suggests similarly named alternatives |
| `KANJECT002` | Info | Information | Valid template placeholder confirmation | Informational only — no action required |
| `KANJECT003` | Warning | Usage | Empty `{}` placeholder in key template | Add a property name inside the braces or use `{{` / `}}` for literal braces |
| `KANJECT1001` | Warning | Usage | `WithIndexAsync` / collection methods require `CommitAsync()` | Call `Repository.CommitAsync()` after the write |
| `KANJECT1002` | Warning | Usage | Transactional methods must be wrapped in `BeginTransaction()` … `CommitAsync()` | Add `Repository.BeginTransaction()` before and `Repository.CommitAsync()` after |

> **Suppression:** Annotate the enclosing method with `[CommitsTransaction]` or add a `// no-commit-required` comment next to `BeginTransaction()` to suppress KANJECT1001/KANJECT1002.

## Transaction Cheat-Sheet

DynamoDB writes through Kanject repositories are batched in a transaction. The analyzer rules `KANJECT1001` and `KANJECT1002` enforce this pattern at compile time.

### Required Pattern

```csharp
repository.BeginTransaction();

// … one or more write operations …

await repository.CommitAsync();
```

### Methods That Require Transactions

| Method Pattern | Why |
| --- | --- |
| Any method ending in `WithIndexAsync` | Writes the main item **and** updates GSI projection items atomically |
| `InsertItemCollectionAsync` | Inserts multiple entities sharing a partition key in a single transaction |
| `AddOrUpdateItemCollectionAsync` | Upserts all entities in a collection atomically |
| `UpsertOrRemoveItemCollectionAsync` | Upserts new items and removes stale ones atomically |
| `RemoveItemCollectionAsync` | Deletes all entities in a collection atomically |

## Key Template Validation

The `KANJECT001` / `KANJECT003` analyzers validate `{Placeholder}` expressions inside key template strings at compile time. If a placeholder references a property that doesn't exist on the entity, the build fails with a suggestion for the closest match.

The following attributes support `{PropertyName}` template interpolation:

| Attribute | Template Parameter Position | Description |
| --- | :---: | --- |
| `[KeyTemplate]` | 1st argument | Primary hash/sort key template |
| `[ShardedKeyTemplate]` | 2nd argument | Key template with automatic shard distribution |
| `[Collection]` | 2nd & 3rd arguments | Hash and range templates for collection keys |
| `[Range]` | 2nd argument | Range key template for composite sort keys |
| `[Searchable]` | 2nd argument | Template for searchable GSI key projection |
| `[AggregateCollectionCount]` | 2nd argument | Counter key template for collection-level aggregates |
| `[AggregateCount]` | 2nd argument | Counter key template for entity-level aggregates |
| `[AggregateSum]` | 2nd argument | Sum key template for entity-level aggregates |

### Template Syntax

```
// Valid: references entity property "UserId"
[KeyTemplate("USER#{UserId}")]

// Invalid: "UserIdentifier" doesn't exist → KANJECT001 error
[KeyTemplate("USER#{UserIdentifier}")]

// Warning: empty placeholder → KANJECT003
[KeyTemplate("USER#{}")]

// Literal braces (no interpolation)
[KeyTemplate("DATA#{{literal}}")]
```
